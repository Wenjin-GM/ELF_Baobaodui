# 智能安全工具柜 PyQt5 触摸屏界面框架大纲

> 项目：宝宝队 ELF 2 / RK3588 智能安全工具柜  
> 界面目标：先完成可演示、可滑动换页、可点击跳转的前端框架，后续再接入板端外设、传感器、视觉识别结果等数据。

---

## 1. 界面定位

本界面运行在 ELF 2 开发板 Ubuntu 22.04 桌面环境上，面向前门触摸屏操作。

当前项目结构中，前门方案主要包含：

- 触摸显示屏：使用 `显示屏/` 目录图片中的 HDMI 接口 10 寸触摸屏，安装在工具柜前门外侧。
- 柜外认证：PN532 NFC、人脸识别 USB 摄像头。
- 柜内识别：OV13855 / MIPI CSI 摄像头 + YOLO 工具检测模型。
- 环境控制：SHT30 温湿度传感器、风扇继电器。
- 执行控制：电磁锁、一区电池盒充电控制、声光报警器、电风扇。当前阶段除湿和降温都通过开启电风扇实现。

界面第一版按 HDMI 10 寸触摸屏做基准布局。程序启动后全屏运行，并自动读取实际分辨率，再按比例缩放页面间距、字号和摄像头预览区域。

## 2. 总体交互方式

### 2.1 页面切换

支持两种切换方式：

| 交互方式 | 实现方式 | 用途 |
|---|---|---|
| 点击底部导航栏 | 按钮触发 `setCurrentIndex()` | 稳定、适合工业触摸屏 |
| 左右滑动换页 | 监听触摸拖动距离，或使用 `QSwipeGesture` | 快速切换总览/工具/环境/记录 |

### 2.2 触摸屏设计原则

- 所有主要按钮高度不小于 `56 px`，避免误触。
- 底部导航栏固定显示，便于快速跳转。
- 关键报警状态使用颜色块、图标、闪烁边框，不依赖小字说明。
- 页面不要堆太多文字，重点数据用大字号显示。
- 工业现场光线复杂，背景建议使用深色或浅灰工作台风格，状态色保持统一。

### 2.3 推荐导航结构

```text
┌──────────────────────────────────────────────┐
│ 顶部状态栏：系统名 / 时间 / 网络 / NPU / 登录状态 │
├──────────────────────────────────────────────┤
│                                              │
│                             主页面区          │
│                                              │
├──────────────────────────────────────────────┤
│ 总览 | 认证 | 工具 | 充电 | 环境 | 记录 | 设置 │
└──────────────────────────────────────────────┘
```

底部导航栏不是固定显示全部页面，而是由当前系统状态动态生成。当前状态下不能显示的页面，不出现在底部导航栏中，也不能通过左右滑动进入。

## 3. 系统状态机与页面显示逻辑

界面显示由系统状态机驱动。每个状态都必须明确：

```text
当前状态 -> 默认显示页面 -> 底部导航显示页面 -> 允许操作 -> 操作后切换状态
```

页面在当前状态下只有“可显示”和“不可显示”两种结果，不使用“部分显示”等模糊规则。

### 3.1 状态与默认页面

| 状态 | 含义 | 默认页面 | 底部导航显示 |
|---|---|---|---|
| `STANDBY` | 未认证待机 | 总览页 | 总览、认证 |
| `AUTH_PENDING` | 正在认证 | 认证页 | 总览、认证 |
| `USER_AUTHED` | 普通用户已认证 | 总览页 | 总览、工具、充电、环境、记录、认证 |
| `ADMIN_AUTHED` | 管理员已认证 | 总览页 | 总览、认证、工具、充电、环境、记录、设置 |
| `CABINET_OPEN` | 已开锁或柜门打开 | 工具页 | 总览、工具、充电、环境、记录 |
| `CHECKING_AFTER_CLOSE` | 关门后盘点中 | 工具页 | 总览、工具 |
| `ALARM_ACTIVE` | 报警状态 | 异常对应页，无法判断时显示总览页 | 总览、工具、充电、环境、记录、认证 |
| `MAINTENANCE` | 调试维护模式 | 调试页 | 调试、设置、总览 |
| `OFFLINE` | 后端或关键硬件离线 | 总览页 | 总览、认证、记录 |

### 3.2 状态跳转逻辑

| 当前状态 | 当前页面 | 操作 | 成功后状态 | 成功后默认页面 |
|---|---|---|---|---|
| `STANDBY` | 认证页 | 普通用户刷卡或人脸认证成功 | `USER_AUTHED` | 总览页 |
| `STANDBY` | 认证页 | 管理员认证成功 | `ADMIN_AUTHED` | 总览页 |
| `STANDBY` | 认证页 | 认证失败 | `STANDBY` | 认证页 |
| `AUTH_PENDING` | 认证页 | 认证成功 | `USER_AUTHED` 或 `ADMIN_AUTHED` | 总览页 |
| `AUTH_PENDING` | 认证页 | 取消认证或认证超时 | `STANDBY` | 总览页 |
| `USER_AUTHED` | 总览页或认证页 | 申请开柜成功 | `CABINET_OPEN` | 工具页 |
| `ADMIN_AUTHED` | 总览页或认证页 | 申请开柜成功 | `CABINET_OPEN` | 工具页 |
| `CABINET_OPEN` | 工具页 | 检测到柜门关闭并上锁 | `CHECKING_AFTER_CLOSE` | 工具页 |
| `CHECKING_AFTER_CLOSE` | 工具页 | 盘点正常 | `STANDBY` | 总览页 |
| `CHECKING_AFTER_CLOSE` | 工具页 | 盘点发现缺失或错放 | `ALARM_ACTIVE` | 工具页 |
| `USER_AUTHED` | 认证页 | 退出登录或超时 | `STANDBY` | 总览页 |
| `ADMIN_AUTHED` | 认证页 | 退出登录或超时 | `STANDBY` | 总览页 |
| `ADMIN_AUTHED` | 设置页 | 进入调试维护 | `MAINTENANCE` | 调试页 |
| `MAINTENANCE` | 设置页或调试页 | 退出维护模式 | `ADMIN_AUTHED` | 设置页 |
| 任意业务状态 | 任意页 | 关键报警产生 | `ALARM_ACTIVE` | 异常对应页 |
| `ALARM_ACTIVE` | 认证页 | 管理员认证成功 | `ADMIN_AUTHED` | 总览页 |
| `ALARM_ACTIVE` | 异常对应页 | 管理员清除报警成功 | `ADMIN_AUTHED` 或 `STANDBY` | 总览页 |
| 任意状态 | 后端检测 | 关键硬件离线 | `OFFLINE` | 总览页 |
| `OFFLINE` | 后端检测 | 关键硬件恢复 | `STANDBY` | 总览页 |

### 3.3 底部导航与滑动规则

底部导航按当前状态生成：

```python
NAV_BY_STATE = {
    "STANDBY": ["总览", "认证"],
    "AUTH_PENDING": ["总览", "认证"],
    "USER_AUTHED": ["总览", "工具", "充电", "环境", "记录", "认证"],
    "ADMIN_AUTHED": ["总览", "认证", "工具", "充电", "环境", "记录", "设置"],
    "CABINET_OPEN": ["总览", "工具", "充电", "环境", "记录"],
    "CHECKING_AFTER_CLOSE": ["总览", "工具"],
    "ALARM_ACTIVE": ["总览", "工具", "充电", "环境", "记录", "认证"],
    "MAINTENANCE": ["调试", "设置", "总览"],
    "OFFLINE": ["总览", "认证", "记录"],
}
```

左右滑动换页只在当前状态的导航列表内循环。例如 `STANDBY` 状态下只能在“总览”和“认证”之间切换，不会滑动到工具、充电、环境、记录、设置页面。

### 3.4 危险操作状态约束

危险操作必须由当前状态、当前页面和用户权限共同决定。

| 操作 | 允许状态 | 允许页面 | 成功后状态 |
|---|---|---|---|
| 普通开柜 | `USER_AUTHED`、`ADMIN_AUTHED` | 总览页、认证页 | `CABINET_OPEN` |
| 临时开锁 | `ADMIN_AUTHED` | 设置页或认证页 | `CABINET_OPEN` |
| 手动开启或关闭充电 | `ADMIN_AUTHED` | 充电页 | 保持 `ADMIN_AUTHED` |
| 禁用自动充电 | `ADMIN_AUTHED` | 充电页、设置页 | 保持 `ADMIN_AUTHED` |
| 修改环境阈值 | `ADMIN_AUTHED` | 设置页 | 保持 `ADMIN_AUTHED` |
| 清除报警 | `ALARM_ACTIVE` 且管理员认证成功 | 异常对应页 | `ADMIN_AUTHED` 或 `STANDBY` |
| 进入调试维护 | `ADMIN_AUTHED` | 设置页 | `MAINTENANCE` |
| 退出调试维护 | `MAINTENANCE` | 调试页、设置页 | `ADMIN_AUTHED` |

危险操作建议使用二次确认弹窗或长按确认，并写入记录。

## 4. 页面规划

## 4.1 总览页 DashboardPage

首页应作为开机后第一屏，展示系统是否安全、工具是否齐全、环境是否合规。

### 页面内容

| 区域 | 显示内容 | 数据来源 |
|---|---|---|
| 五区工具状态 | 1 区充电区、2-5 区普通工具区，显示正常/缺失/错放 | YOLO 工具检测结果 |
| 柜门状态 | 已关门 / 已开门 / 异常未关 | 电磁锁、门磁或后端状态 |
| 认证状态 | 当前用户、认证方式、最近一次认证时间 | NFC / 人脸识别 |
| 环境卡片 | 温度、湿度、风扇状态、当前风扇用途 | SHT30 + 风扇继电器 |
| 充电概览 | 1 区电池盒是否在位、是否充电、是否异常 | 充电控制模块 + 工具检测结果 |
| 报警条 | 工具缺失、错放、湿度超限、非法刷卡 | 后端事件总线 |

### 推荐布局

```text
┌────────────────────────────┬────────────────────┐
│ 五区工具状态图              │ 环境状态             │
│ 1区 2区 3区                 │ 25.6℃  52%RH        │
│ 4区 5区                     │ 风扇：开 / 用途：除湿  │
├────────────────────────────┼────────────────────┤
│ 1区电池盒充电状态            │ 最近事件 / 当前用户   │
└────────────────────────────┴────────────────────┘
```

### 状态颜色

| 状态 | 颜色建议 |
|---|---|
| 正常 / 在位 | 绿色 |
| 待归还 / 离位 | 黄色 |
| 错放 / 非授权 / 超限 | 红色 |
| 未连接 / 未启用 | 灰色 |
| 充电中 | 蓝色 |

## 4.4 充电管理页 ChargingPage

用于展示 1 区电池盒充电状态。当前硬件方案只有 1 区需要充电，2-5 区为普通工具区，不在界面上设计多路充电控制。

电池盒逻辑：

- 每次关门并且电磁锁锁上时，系统先给 1 区电池盒通电检测。
- 检测 4 个电池位置是否有电池。
- 每个电池位只显示“在位”或“空位”。
- 如果充电检测模块离线或状态字无效，页面显示“充电检测模块离线”，不更新 4 个电池位。
- 继电器状态只显示“充电电源：开启 / 关闭”。

### 页面内容

| 显示项 | 说明 |
|---|---|
| 1 区电池盒卡片 | 显示电池盒是否在位、继电器是否通电、当前判断结果 |
| 4 个电池位表格 | 每个位置显示在位 / 空位 |
| 充电策略 | 自动 / 手动 / 禁用 |
| 异常提示 | 未检测到电池盒、充电检测模块离线、状态字无效、继电器异常 |
| 操作按钮 | 手动切换通电、禁用自动、恢复自动 |

### 充电位显示规则

| 后端状态 | 页面显示 |
|---|---|
| 模块在线，slot bit = 1 | 在位 |
| 模块在线，slot bit = 0 | 空位 |
| 模块离线 / 状态字无效 | 显示“充电检测模块离线”，不更新电池位 |
| 继电器通电 | 充电电源：开启 |
| 继电器断电 | 充电电源：关闭 |

## 4.5 环境监控页 EnvironmentPage

用于展示 SHT30 温湿度、风扇状态和阈值。当前阶段没有单独的除湿器或制冷模块，除湿和降温都通过电风扇完成。

### 页面内容

- 当前温度：例如 `25.6 ℃`
- 当前湿度：例如 `52 %RH`
- 合规区间：温度 `5-40 ℃`，湿度 `≤ 60 %RH`
- 执行器状态：风扇、报警器。
- 最近 24 小时温湿度曲线。
- 手动控制：风扇开关、恢复自动。

### 当前项目默认阈值

`env_nfc_relay_control.py` 中当前风扇逻辑：

- 湿度 `> 55 %RH`：风扇开启。
- 湿度 `<= 50 %RH`：风扇关闭。

申报书里的目标阈值：

- 温度：`5-40 ℃`
- 湿度：`≤ 60 %RH`

温度高于上限时，当前方案同样开启风扇，界面显示为“风扇降温”。

界面设置页允许管理员修改这些阈值。

## 5. 前后端连接方式

### 5.1 第一阶段：MockBackend

先用模拟数据跑通界面：

- 每 1 秒更新温湿度。
- 每 2 秒随机更新工具状态。
- 点击“刷卡成功”模拟开锁。
- 点击“立即盘点”模拟 YOLO 结果。
- 点击 1 区电池盒卡片模拟充电开启/关闭。

这样不用接板子，也能先把显示屏交互做完。

### 5.2 第二阶段：HardwareBackend

后续接板子时，把 `MockBackend` 替换为 `HardwareBackend`。界面只接收后端状态，不直接阻塞式读取硬件。

HardwareBackend 至少需要向 UI 提供：

| 状态类型 | 字段 |
|---|---|
| 环境状态 | 温度、湿度、风扇状态、报警器状态 |
| 认证状态 | 用户姓名、权限、认证方式、认证结果、开锁结果 |
| 工具状态 | 每区登记数量、当前数量、再借数量、缺失/错放状态 |
| 充电状态 | 电池盒在位、4 个电池位在位布尔值、继电器状态、模块在线状态、状态字有效性 |
| 系统状态 | 当前状态机状态、在线/离线、报警状态 |
| 记录事件 | 认证、开柜、盘点、报警、环境变化、充电动作 |

推荐信号：

```python
env_updated = pyqtSignal(dict)
auth_updated = pyqtSignal(dict)
tools_updated = pyqtSignal(dict)
charging_updated = pyqtSignal(dict)
system_state_changed = pyqtSignal(str)
event_added = pyqtSignal(dict)
alarm_raised = pyqtSignal(dict)
```

充电状态中的 4 个电池位字段应为布尔值：

```python
charging_state = {
    "module_online": True,
    "status_valid": True,

    "relay_on": False,
    "slots": [True, False, True, True]

```

## 6. 界面首版开发顺序

1. 建立 `QMainWindow + QStackedWidget + 底部导航栏`。
2. 实现状态机、默认页面切换和底部导航动态生成。
3. 做总览页静态布局，填入模拟数据。
4. 加入左右滑动换页和点击跳转，滑动范围限制在当前状态可显示页面内。
5. 做认证页、工具页、充电页、环境页、记录页、设置页、调试页的空框架。
6. 增加 `MockBackend`，每秒发信号刷新界面。
7. 加入 log 或 SQLite 写入接口，但第一版可以只写关键事件。
8. 接入板端 `SHT30` 温湿度读取。
9. 接入 `PN532` 授权卡状态。
10. 接入人脸识别结果。
11. 接入 YOLO 工具检测和 1 区电池盒充电状态。

## 7. 首版界面必须显示的信息清单

为了比赛演示，第一版至少要在屏幕上看到：

- 当前系统状态：未认证待机 / 普通用户已认证 / 管理员已认证 / 柜门打开 / 盘点中 / 报警 / 离线 / 维护模式。
