基于 ROS2 的工作体系框架

一、总体设计原则
1. 高风险执行动作不使用纯 topic：
   - 门锁、风扇、蜂鸣器等执行器用 service/action。
   - 调用方需要得到成功/失败结果。
   - 门锁单次通电时间限制必须写在 actuator_node 内部，不完全信任外部参数。

2. 高算力任务采用触发式：
   - NFC、人脸识别只在认证窗口内工作。
   - 柜内视觉盘点只在“柜门关闭后”或“用户点击立即盘点”时触发。
   - 柜内摄像头图像不高频发布原图，只在每次触发盘点时发布一次图像，并保存一份本地图片。


二、节点划分

1. ui_node（显示屏节点）

类型：发布和订阅节点，必要时调用 service/action。

职责：
1. 显示系统状态、当前用户、温湿度、工具盘点结果、充电状态、事件日志。
2. 接收用户点击或触摸操作，例如：
   - 申请开柜
   - 立即盘点
   - 手动开/关风扇
   - 临时开锁
   - 进入/退出维护模式
3. 向 cabinet_logic_node 发送用户请求。
4. 不直接订阅传感器、电池盒、执行器等底层原始 topic，避免 UI 重复承担业务判断。
5. 只订阅 cabinet_logic_node 汇总后的、面向界面显示的数据。

订阅：
1. /ui/summary：总览页聚合状态，包括系统状态、当前用户、环境摘要、工具摘要、充电摘要、报警摘要。
2. /ui/environment：环境页聚合状态，包括温度、湿度、风扇状态、报警状态、阈值、趋势提示。
3. /ui/inventory：工具页聚合状态，包括五区盘点结果、缺失/错放列表、最近盘点时间。
4. /ui/inventory_image：最近一次盘点图像或压缩图像。
5. /ui/battery：充电页聚合状态，包括电池盒状态、四个电池位状态、充电开关状态、模块在线状态。
6. /ui/auth：认证页聚合状态，包括 NFC/人脸认证进度、认证结果、当前用户。
7. /ui/events：系统事件日志。

调用：
1. /cabinet/request_open：用户点击“申请开柜”。
2. /cabinet/request_inventory：用户点击“立即盘点”。
3. /cabinet/request_manual_fan：管理员手动开/关风扇。
4. /cabinet/request_temp_unlock：管理员临时开锁。


2. cabinet_logic_node（系统主控逻辑节点）

类型：发布、订阅、service/action 客户端和 service 服务端。

职责：
1. 维护系统状态机：
   - STANDBY：未认证待机
   - AUTH_PENDING：正在认证
   - USER_AUTHED：普通用户已认证
   - ADMIN_AUTHED：管理员已认证
   - CABINET_OPEN：柜门打开
   - CHECKING_AFTER_CLOSE：关门后盘点中
   - ALARM_ACTIVE：报警状态
   - MAINTENANCE：维护模式
   - OFFLINE：关键硬件离线
2. 处理认证流程：
   - 收到“申请开柜”后，同时启动 NFC 认证和人脸认证。
   - 认证窗口建议为 5s。
   - NFC 或人脸任一通过即可开柜。
   - 超时则认证失败，回到待机或认证页提示失败。
3. 处理开锁流程：
   - 认证通过后调用 actuator_node 的开锁 service/action。
   - 开锁成功后进入 CABINET_OPEN。
4. 处理盘点流程：
   - 柜门关闭后自动触发视觉盘点。
   - 用户点击“立即盘点”时手动触发视觉盘点。
   - 盘点正常则回到 STANDBY 或已认证状态。
   - 发现缺失、错放等异常则进入 ALARM_ACTIVE，并触发蜂鸣器。
5. 处理环境控制：
   - 订阅 /env/state。
   - 温湿度超限时请求 actuator_node 开风扇。
   - 环境恢复后请求 actuator_node 关风扇。
6. 汇总面向 UI 的显示状态：
   - 将 /env/state、/battery/state、/actuator/state、/vision/inventory_result、/auth/result 等底层数据整理成 /ui/* topic。
   - UI 只关心“该显示什么”，不直接判断底层传感器是否异常、风扇是否该开、是否该报警。
7. 统一发布系统事件：
   - 认证成功/失败
   - 开柜/关柜
   - 盘点开始/完成
   - 工具缺失/错放
   - 温湿度超限
   - 执行器动作结果

发布：
1. /cabinet/state：系统状态。
2. /cabinet/events：事件日志。
3. /auth/result：认证结果汇总，主要供记录和调试，也可被 cabinet_logic_node 内部转换为 /ui/auth。
4. /ui/summary：总览页聚合状态。
5. /ui/environment：环境页聚合状态。
6. /ui/inventory：工具页聚合状态。
7. /ui/inventory_image：最近一次盘点图像或压缩图像。
8. /ui/battery：充电页聚合状态。
9. /ui/auth：认证页聚合状态。
10. /ui/events：面向 UI 的事件日志。

订阅：
1. /env/state：温湿度数据。
2. /battery/state：电池盒状态。
3. /actuator/state：执行器状态。
4. /vision/inventory_result：柜内视觉盘点结果。
5. /vision/inventory_image：最近一次盘点图像或压缩图像。
6. 后续如增加门磁，可订阅 /door/state。

提供服务：
1. /cabinet/request_open
2. /cabinet/request_inventory
3. /cabinet/request_manual_fan
4. /cabinet/request_temp_unlock

调用：
1. nfc_node 的 ReadNfcCard action。
2. face_node 的 AuthenticateFace action。
3. vision_node 的 RunInventory action。
4. actuator_node 的 open_lock / set_fan / beep service。


3. nfc_node（NFC 节点）

类型：action 服务端，底层按需轮询 PN532。

硬件：
1. PN532 接 SDA.0/SCL.0。
2. 当前板端验证为 /dev/i2c-7。
3. I2C 地址 0x24。

工作方式：
1. 默认空闲，不持续刷卡。
2. cabinet_logic_node 发起 ReadNfcCard action 后，进入刷卡窗口。
3. 在 timeout 时间内低频轮询 PN532。
4. 检测到卡片后读取 UID。
5. 查询授权卡数据库，返回：
   - success
   - uid
   - authorized
   - user_name
   - role
6. 成功或超时后自动退出工作状态。
7. 如果未来连接 PN532 IRQ 引脚，可将底层轮询改为 GPIO 中断触发。

建议 action：
ReadNfcCard
目标：
  timeout_sec
结果：
  success
  uid
  authorized
  user_name
  role
  message


4. face_node（人脸识别与验证节点）

类型：action 服务端。

硬件：
1. 柜外 USB 摄像头，当前倾向 /dev/video21。
2. 使用 USB/face_auth 中的 InsightFace 识别流程。

工作方式：
1. 默认空闲，不持续做人脸识别。
2. cabinet_logic_node 发起 AuthenticateFace action 后，打开认证窗口。
3. 在 timeout 时间内采集图像并做人脸识别。
4. 检测到授权人员后返回成功。
5. 成功或超时后自动退出工作状态。

建议 action：
AuthenticateFace
目标：
  timeout_sec
  min_confidence
结果：
  success
  user_name
  role
  confidence
  message


5. env_node（温湿度传感器节点）

类型：纯发布节点。

硬件：
1. SHT30 接 SDA.1/SCL.1。
2. 当前板端验证为 /dev/i2c-4。
3. I2C 地址 0x44。

工作方式：
1. 常驻运行。
2. 每 1s 读取一次 SHT30。
3. 读取成功后立即发布温湿度数据。
4. 消息中带 timestamp，不需要人为设计“读取与输出相差 0.5s”。

发布：
/env/state
字段建议：
  temperature
  humidity
  valid
  timestamp


6. vision_node（柜内视觉识别节点）

类型：action 服务端，必要时发布一次图像和一次识别结果。

硬件：
1. 柜内摄像头当前优先使用 /dev/video11。
2. 后续接 YOLO/RKNN 工具识别模型。

工作方式：
1. 默认空闲，不持续推理，不高频发布原图。
2. 以下两种情况触发盘点：
   - 柜门关闭后，cabinet_logic_node 自动调用 RunInventory。
   - 用户在显示屏点击“立即盘点”，ui_node 请求 cabinet_logic_node，再由 cabinet_logic_node 调用 RunInventory。
3. 每次触发后：
   - 拍摄一张柜内图像。
   - 将原图保存到本地，例如：
     data/inventory_images/YYYYMMDD_HHMMSS.jpg
   - 发布一次 /vision/inventory_image，供显示屏展示。
   - 调用视觉模型进行识别。
   - 发布一次 /vision/inventory_result。
   - 将识别结果写入本地记录或数据库。
4. 识别结果应包含：
   - 每种工具数量。
   - 五区各自状态。
   - 缺失、错放、异常列表。
   - 每个目标的类别、置信度、坐标位置。

建议 action：
RunInventory
目标：
  reason              # door_closed / user_request / debug
  save_image
结果：
  success
  image_path
  is_normal
  zones
  detections
  message

发布：
1. /vision/inventory_image：每次盘点发布一张图像或压缩图像，供 cabinet_logic_node 汇总后转发为 /ui/inventory_image。
2. /vision/inventory_result：每次盘点发布一次结构化识别结果。


7. battery_node（电池盒充电状态节点）

类型：纯发布节点，当前可先作为占位节点。

说明：
1. 电池充电模块目前尚未连接。
2. 初期可发布 mock 或 unknown 状态，保证显示屏和主控逻辑接口先稳定。
3. 等电池充电模块接入后，再替换为真实读取逻辑。

发布：
/battery/state
字段建议：
  module_online
  status_valid
  box_present
  slots             # 四个电池位：present / empty / unknown
  battery_levels    # 若硬件支持电量，则发布百分比；否则为空或 unknown
  charging_on
  timestamp


8. actuator_node（继电器/执行器控制节点）

类型：service/action 服务端。

硬件：
1. 风扇：GPIO.28 -> gpiochip3 line 9 -> 二号继电器，低电平有效。
2. 门锁：GPIO.25 -> gpiochip3 line 3 -> 三号继电器，低电平有效。
3. 蜂鸣器：GPIO.23 -> gpiochip3 line 4，触发电平待实测。

工作方式：
1. 常驻运行。
2. 默认把所有继电器置于安全状态。
3. 接收控制请求后执行，并返回结果。
4. 门锁通电保护必须写死在节点内部：
   - 单次通电时间不得超过 0.5s。
   - 默认建议 0.3s。
   - 即使外部传入更长时间，也必须截断或拒绝。
5. 风扇可支持 set true / false。
6. 蜂鸣器支持 beep(times, interval)。

提供服务：
1. /actuator/open_lock
   请求：
     pulse_sec
   响应：
     success
     actual_pulse_sec
     message

2. /actuator/set_fan
   请求：
     on
     reason
   响应：
     success
     message

3. /actuator/beep
   请求：
     times
     on_sec
     off_sec
   响应：
     success
     message

发布：
/actuator/state
字段建议：
  lock_busy
  fan_on
  buzzer_busy
  timestamp


三、核心业务流程

1. 用户申请开柜流程

1. 用户在显示屏点击“申请开柜”。
2. ui_node 调用 /cabinet/request_open。
3. cabinet_logic_node 进入 AUTH_PENDING。
4. cabinet_logic_node 同时启动：
   - nfc_node: ReadNfcCard(timeout=5s)
   - face_node: AuthenticateFace(timeout=5s)
5. NFC 或人脸任意一个认证成功：
   - cabinet_logic_node 取消另一个认证任务。
   - 发布认证成功事件。
   - 调用 /actuator/open_lock。
6. 开锁成功：
   - 状态切换为 CABINET_OPEN。
   - ui_node 显示用户身份、开柜成功、柜门已打开。
7. 5s 内两种认证均未成功：
   - 状态回到 STANDBY 或保持认证页提示失败。
   - 发布认证失败事件。


2. 柜门关闭后自动盘点流程

1. 系统检测到柜门关闭。
   - 当前如果没有门磁，可先由用户点击“完成操作/开始盘点”模拟。
   - 后续增加门磁后订阅 /door/state。
2. cabinet_logic_node 进入 CHECKING_AFTER_CLOSE。
3. cabinet_logic_node 调用 vision_node 的 RunInventory(reason=door_closed)。
4. vision_node 拍摄一张柜内图像、保存、发布一次图像、运行模型、发布识别结果。
5. cabinet_logic_node 根据识别结果判断：
   - 正常：回到 STANDBY，记录盘点正常。
   - 异常：进入 ALARM_ACTIVE，调用 /actuator/beep 让蜂鸣器响 3 下。
6. ui_node 显示最新盘点图片、五区结果和异常提示。
   - ui_node 实际订阅的是 /ui/inventory 和 /ui/inventory_image。
   - /vision/inventory_result 和 /vision/inventory_image 先进入 cabinet_logic_node，由主控节点汇总、判断和转发。


3. 用户点击立即盘点流程

1. 用户在工具页点击“立即盘点”。
2. ui_node 调用 /cabinet/request_inventory。
3. cabinet_logic_node 判断当前状态是否允许盘点。
4. 若允许，调用 vision_node 的 RunInventory(reason=user_request)。
5. vision_node 每次只发布一次图像和一次识别结果，并保存图片和结果。
6. ui_node 刷新工具页和总览页显示。
   - ui_node 根据 /ui/inventory、/ui/summary、/ui/inventory_image 刷新，不直接订阅 vision_node 的原始结果。


4. 温湿度自动控制流程

1. env_node 每 1s 发布 /env/state。
2. cabinet_logic_node 订阅温湿度。
3. 湿度或温度超过阈值时：
   - cabinet_logic_node 调用 /actuator/set_fan(on=true, reason=除湿/降温)。
   - 发布环境异常事件。
4. 环境恢复到关闭阈值后：
   - cabinet_logic_node 调用 /actuator/set_fan(on=false, reason=环境恢复)。
   - 发布环境恢复事件。
5. cabinet_logic_node 将环境数据、风扇状态、报警状态汇总为 /ui/environment 和 /ui/summary。
6. ui_node 显示温湿度、风扇状态和事件，但不直接订阅 /env/state。

五、当前阶段实施建议

第一步：先做最小 ROS2 骨架
1. env_node：发布真实 SHT30 温湿度。
2. actuator_node：提供真实风扇、门锁、蜂鸣器服务。
3. cabinet_logic_node：订阅 /env/state，调用 /actuator/set_fan，并发布 /ui/environment。
4. ui_node：先订阅 /ui/environment，显示温湿度；按钮调用 /cabinet/request_manual_fan。

第二步：做认证链路
1. nfc_node：实现 ReadNfcCard action。
2. face_node：实现 AuthenticateFace action。
3. cabinet_logic_node：实现 request_open 流程。

第三步：做触发式视觉盘点
1. vision_node：实现 RunInventory action 的拍照、保存和一次图像发布。
2. 后续接入 YOLO/RKNN 模型，替换 mock 检测结果。

第四步：显示屏完整接入
1. 将 show/Script 的 MockBackend 替换为 RosBackend。
2. 页面层尽量不改，只让 RosBackend 保持与 MockBackend 相同的 Qt 信号接口。
3. 用户点击操作由 RosBackend 转成 ROS service/action 请求。
