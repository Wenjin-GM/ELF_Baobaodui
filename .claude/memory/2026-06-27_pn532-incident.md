# 2026-06-27 PN532 Incident Memory

## Summary

The PN532 module became unstable during ROS/NFC integration experiments. The user requested full restoration of the NFC module code.

## What Happened

Initial known-good standalone NFC test:

```bash
python3 PN532/tests/test_campus_card_uid.py --bus 7 --timeout 30
```

This once read campus card UID `19517D20`.

During ROS integration:

- A continuous NFC polling implementation was introduced earlier and later removed.
- `nfc_node` was changed to lazy-open PN532 only on action request.
- Experiments with `InAutoPoll` and alternate safe scripts were added, then removed.
- No-card or failed-card `InListPassiveTarget` tests caused PN532 to stop ACKing on I2C.
- When PN532 stops ACKing, `i2cdetect -y 7` shows all `--`.
- Since PN532 reset pin is not wired to GPIO, recovery requires power-cycling the PN532 module.

Important correction:

- When PN532 appeared to recover earlier, it was actually because the user manually power-cycled the module. Do not assume automatic recovery.

## Current Restored State

Restored files:

- `PN532/drivers/i2c_pn532.py` restored to original-style driver API:
  ```python
  nfc.read_passive_target_id(timeout=1.0)
  ```
- Temporary files removed:
  - `PN532/tests/test_safe_uid_once.py`
  - `PN532/tests/test_card_present_uid.py`
  - `docs/PN532_SAFE_USE.md`
- `nfc_node.py` restored to call:
  ```python
  self.nfc.read_passive_target_id(timeout=timeout)
  ```
  with no `safe=` argument.

Board rebuild completed after restoring `nfc_node`.

## Remaining Cache Issue

A root-owned pycache may remain on board:

```text
~/smart_tool_cabinet/PN532/drivers/__pycache__/i2c_pn532.cpython-310.pyc
```

User should remove it on board:

```bash
cd ~/smart_tool_cabinet
sudo rm -rf PN532/drivers/__pycache__ PN532/tests/__pycache__
```

Then power-cycle PN532 and confirm:

```bash
i2cdetect -y 7
```

Expected healthy PN532:

```text
20: -- -- -- -- 24 -- -- ...
```

## Retest Baseline

After cache removal and PN532 power-cycle:

```bash
cd ~/smart_tool_cabinet/PN532
python3 tests/test_campus_card_uid.py --bus 7 --timeout 30
```

## Rules Going Forward

- Do not add or run new PN532 polling experiments until original standalone test works again.
- Do not run long no-card NFC tests.
- If `0x24` disappears, stop all NFC/ROS processes and ask user to power-cycle PN532.
- Best hardware fix: wire PN532 `RSTPDN/RESET` to a GPIO so software can recover the module.
- ROS integration should wait until standalone baseline is stable.
